# Thinking Tools — 思考ツール

思考プロセス（process.md）で使用する補助ツール群。

## 位置づけ

各ツールは思考プロセスの基本ループ（OODA）の特定フェーズを支援する。どのフェーズでどのツールを使うかを意識することで、ツールの空回りを防ぐ。

| ツール | 対応する OODA フェーズ |
|---|---|
| 問い返しフレーム | Observe / Orient |
| 抽象度の制御 | Orient（層を行き来する解釈補助） |
| モードの合成ルール | Orient の最終段（視点確定） |
| 評価基準 | Act 後の評価 |
| 振り返り | 次の Observe / Orient への学習 |
| 更新ルール | 学習ループの還流 |
| エコーチェンバー・チェック | Orient / Decide（自己強化ループの検知） |
| 反証 | Decide / Act 後の評価（反証条件の定義と検証） |
| 非反証領域 | Orient / Decide（直接反証しにくい要素の扱い） |

---

## 問い返しフレーム（Inquiry Framework）

状況に応じて以下の問いを使い、思考の精度を高めること。
すべて使う必要はない。最も本質に効くものを選択する。

### 1. 前提を疑う

- その前提は本当に正しいか？どの程度確からしいか？
- 暗黙の前提は何か？
- その前提が崩れた場合、どうなるか？

### 2. 目的を明確にする

- これは何を達成するための議論か？
- 成功とはどの状態か？どう測定するか？
- 誰にとっての価値か？

### 3. 構造に落とす

- これは構造的にどういう問題か？（人・プロセス・情報・技術）
- どこがボトルネックか？
- 分解するとどのレイヤーの問題か？
- 指標やKPIを分解式で構造化し、どの変数に介入すべきかを特定できるか？

### 4. 再現性を問う

- それは再現可能か？他の状況でも成立するか？
- 特定の個人や状況に依存していないか？

### 5. トレードオフを明確にする

- 何を得て、何を失うのか？
- 制約条件は何か？（コスト・時間・組織負荷など）
- 他に選択肢はあるか？

### 6. 時間軸で捉える

- 短期と長期でどう違うか？
- 今やるべきか？後でも良いか？
- 判断を更新するタイミングはいつか？

### 7. 実行可能性を確認する

- 明日から何を変えるのか？
- どの単位で試すのか？
- 小さく検証できるか？

## 抽象度の制御（Abstraction Level）

思考の抽象度を意識的に移動させるプロセス。
抽象（原則）・構造（分解）・具体（実行）の3層を行き来し、思考の精度を高める。

### 3つの層

| 層 | 問い | 思考の内容 |
|---|---|---|
| 抽象（原則） | なぜやるのか？何を目指すのか？ | 目的・原則・方針 |
| 構造（分解） | どう分解できるか？何が関係しているか？ | 要素分解・依存関係・優先順位 |
| 具体（実行） | 誰が何をいつやるか？ | アクション・担当・期限・成果物 |

### 移動のルール

**下降（抽象 → 構造 → 具体）**

- 抽象で合意してから構造に降りる。原則が曖昧なまま分解しない
- 構造で整理してから具体に降りる。分解せずにアクションを決めない
- 層を飛ばさない。抽象から直接具体に降りると、構造の欠落を見落とす

**上昇（具体 → 構造 → 抽象）**

- 具体の議論が噛み合わない時は、構造に戻って前提を確認する
- 構造の議論が発散する時は、抽象に戻って目的を再確認する
- 「そもそも何のためにやっているのか」という問いは、上昇のシグナルである

**現在地の宣言**

- 今どの層で思考しているかを意識する
- 層を移動する時は、移動の意図を明示する（「具体に降りると…」「一度原則に戻ると…」）
- 異なる層の議論を同時に混ぜない

### 各層でのチェック

- **抽象（原則）**: core の原則と整合しているか。目的が明確で、成功基準が定義できるか
- **構造（分解）**: 要素の分解に漏れ・重複がないか。依存関係と優先順位が明確か
- **具体（実行）**: アクションが検証可能で、実行者が特定できるか。抽象で定めた目的に対して、この具体が貢献するかを逆算できるか

### アンチパターン

- 抽象だけで終わる（方針は立派だが何も動かない）
- 具体だけで走る（やることは決まったが、なぜやるか不明）
- 構造を飛ばす（原則と実行の間に隙間ができ、判断がぶれる）
- 層を混ぜて議論する（目的の話と手段の話が同時に進み、収束しない）

## モードの合成ルール（Composition Rule）

複数の thinking-mode が該当する場合の選択・合成ルール。

### 基本原則

- 1つの状況に対して、主モードは1つに絞る
- 補助モードとして最大1つを追加できる
- 主モードの出力スタイルとNGルールが優先される

### 主モードの選択基準

| 基準 | 説明 |
|---|---|
| 意思決定者の役割 | 最終的に誰が判断するかに最も近いモードを選ぶ |
| 問題の本質 | 組織問題 → executive、技術問題 → tech-review、成長課題 → one-on-one |
| 受け手の期待 | 受け手が何を求めているかに最も合致するモードを選ぶ |

### 補助モードの適用

- 主モードのフレームで思考しつつ、補助モードの追加ルールから関連する項目のみ取り込む
- 補助モードの出力スタイルは適用しない（主モードの出力スタイルに統一）
- 補助モードのNGルールは参考にするが、主モードのNGが優先

### 合成時の衝突解決

主モードと補助モードの指針が対立する場合:

1. 主モードを優先する（主モードが思考の軸である）
2. それでも判断できない場合は、core/foundation.md の優先順位ルールに従う
3. 優先順位ルールでも衝突する場合は、core/foundation.md の衝突解決に従う

### 合成の典型パターン

| 状況 | 主モード | 補助モード | 理由 |
|---|---|---|---|
| 技術選定の経営判断 | executive | tech-review | 最終判断は経営視点だが、技術的な妥当性も必要 |
| メンバーへのアーキテクチャ説明 | one-on-one | tech-review | 対話のフレームで進めつつ、技術的な正確性を担保 |
| プロダクトの技術的実現性検討 | product | tech-review | プロダクト価値を軸に、技術制約を組み込む |
| 組織課題を持つメンバーとの1on1 | one-on-one | executive | 対話を軸に、組織的な視点を補助として入れる |

## エコーチェンバー・チェック（Echo Chamber Check）

自分の思考OSが自己強化ループに閉じていないかを確認する。

### いつ使うか

- Slow Path に入った重要判断
- 自分の既存原理に強くフィットする判断
- 自分の過去判断を正当化している可能性がある判断
- AIや他者の反応が、自分の思想の補強だけになっていると感じる時

### チェック観点

**1. 別フレームでの解釈** — この問題を、まったく違う価値観で見るとどうなるか？

- 短期最適 / 感情優先 / 個人依存 / 現場都合 / 関係性優先

**2. 反対者シミュレーション** — この意思決定に対して、強く反対する人は何を根拠にするか？

**3. 失敗モード** — この判断が間違っているとしたら、どこが最も危ないか？

**4. 非構造シグナル** — 言語化できていない違和感は何か？

- 現場の空気 / 関係性の歪み / 感情的な抵抗 / 説明しにくい違和感

**5. AIバイアスチェック** — この結論は、自分が欲しい答えをAIに生成させていないか？

## 反証（Falsification）

疑いを単なる態度で終わらせず、反証可能な形にするためのツール。

このツールは support.md の Uncertainty 原則（検証のトリガー条件・代替シグナル・前提が崩れた場合の影響範囲見積もり）を、より積極的な反証フレームとして拡張したもの。新規概念ではなく、既存原則の運用形である。

**1. Kill Condition（撤退条件）** — 何が起きたら、この判断を明確に「間違い」と認めるか？

**2. Disconfirming Evidence（反証材料）** — Kill Condition を満たす兆候・データ・事象は何か？

**3. Opposing Model（反対仮説）** — この判断と真逆のモデルを成立させるとしたらどうなるか？

**4. Survivability Check（生存性チェック）** — それでもこの判断は生き残るか？

残す場合も、捨てる場合も、理由を明示する。

## 非反証領域（Non-falsifiable Domain）

直接的な反証が困難、または不可能な要素を扱うためのツール。

対象例: ビジョン、信頼、文化、美意識、人への期待。
これらは通常の Kill Condition によって扱うと歪むため、以下の方法で観測する。

**1. 代理観測（Proxy Observation）** — 直接ではなく、間接的なシグナルでズレを観測する。

- 信頼: 難しい相談・重要な仕事が自然に集まるか
- 文化: 衝突の質、意思決定の速度、ふるまいの一貫性
- ビジョン: 採用・アサイン・優先順位に一貫して影響しているか

**2. 時間軸評価（Time Horizon Evaluation）** — 短期のノイズで判断せず、中長期での一貫性・変化を見る。

- 短期: ノイズ
- 中期: 傾向
- 長期: 構造

**3. 違和感シグナル（Discomfort Signal）** — 言語化されていない違和感を、重要な観測対象として扱う。即座に判断材料にせず、継続観測の対象にする。

## 評価基準（Evaluation）

出力やプロセスの質を判定する基準。

### 評価の原則

- 出力の「良し悪し」ではなく、思考プロセスの合理性で評価する（意思決定原則）
- 評価基準は事前に明確にする。事後に基準を変えない
- 完璧を求めず、「次に改善すべき点が特定できている」状態を合格とする

### 出力の評価軸

**1. 目的適合性** — 出力が本来の目的に対して適切か。

- 問いに対する答えになっているか
- 受け手（receiver-model）の状態に合った伝え方になっているか
- 過不足なく、必要な情報が含まれているか

**2. 構造の健全性** — 思考の構造が堅牢か。

- 前提が明示されているか。不確実な前提に確信度が伴っているか
- トレードオフが認識・表現されているか
- 時間軸が区別されているか（短期/長期の混同がないか）

**3. 実行可能性** — 出力が具体的なアクションにつながるか。

- 次に何をすべきかが明確か
- 誰がいつどう動くかが特定できるか
- 小さく検証可能な単位になっているか

**4. 再現性** — 同じ状況で、同じ思考プロセスを他者が辿れるか。

- 判断の根拠が属人的でなく、追跡可能か
- 特定の個人・状況への依存がないか

### 評価しないこと

- 結果の成否（判断時点で制御不能な外部要因は評価に含めない）
- 網羅性（全てをカバーすることより、重要な点を押さえているかを重視）

## 振り返り（Reflection）

思考プロセスの振り返りと、システムへのフィードバック還流。

評価基準が**出力の質**（目的適合性・構造の健全性・実行可能性・再現性）を評価するのに対し、振り返りは**思考プロセス自体**（モード選択・トリガー判断・原則の適用）を振り返る。出力の評価を先に行い、その結果をプロセスの振り返りに活かす。

### いつ振り返るか

- Slow Path を経た判断の後（推奨）
- Fast Path でも「何か引っかかる」感覚があった時
- 判断の結果が出た後（成功・失敗を問わず）

全ての判断に振り返りを要求しない。ただし振り返りを習慣化することで、Fast Path の精度が長期的に向上する。

### 振り返りの観点

**1. プロセスの評価** — 判断の「結果」ではなく「プロセス」を振り返る（意思決定原則「意思決定時点の合理性で評価する」）。

- 適切なモードを選択できたか？
- エスカレーション・トリガーは正しく機能したか？（見逃し・過剰反応はなかったか）
- 問い返しフレームの問いは適切だったか？

**2. パターンの検証**

- この状況は既存のusecaseで十分にカバーされていたか？
- 新しいパターンが見つかったか？usecaseとして追加すべきか？
- 既存のusecaseに修正すべき点があったか？

**3. 原則の検証**

- core の原則は役に立ったか？不足している観点はなかったか？
- モードの追加ルールは適切だったか？
- anti-patterns に追加すべき新しいパターンはあったか？

### フィードバックの還流先

| 気づき | 還流先 |
|---|---|
| 新しい状況パターンを発見した | usecase に追加 |
| 既存usecaseの不足に気づいた | usecase を更新 |
| モードのルールに改善点を見つけた | thinking-mode / output-mode / receiver-model を更新 |
| エスカレーション・トリガーの過不足 | process.md のトリガーを更新 |
| 原則そのものへの疑問 | core の更新プロトコルに従い検討 |

振り返りの結果を「次の機会に活かそう」で終わらせない。具体的な更新として、システムに反映する。更新の判断基準・プロセス・優先順位はシステム更新のルールに従う。これがMind as Code を静的な参照物ではなく、進化するシステムにする。

## システム更新のルール（Update Rule）

振り返りと評価の結果を、Mind as Code のシステムに反映する際のルール。「気づき」を「更新」に変換するプロセスを定義する。

### 更新の判断基準

全ての気づきが更新に値するわけではない。以下の条件で判断する。

**更新すべき場合:**

- 同じ気づきが2回以上繰り返された（パターンとして定着している）
- 既存の定義では対応できない状況に遭遇した
- 既存の定義が誤った判断を誘導した

**更新を保留すべき場合:**

- 1回限りの特殊な状況に起因する気づき
- 気づきが十分に言語化できていない（まだ曖昧）
- 既存の定義の解釈で対応可能（定義の不足ではなく、適用の問題）

### 更新のプロセス

**1. 影響範囲の特定** — 更新がどのレイヤーに影響するかを確認する（layer-boundary.md に従う）。

- usecase の追加・修正 → 他レイヤーへの影響なし（最も安全）
- thinking-mode / output-mode / receiver-model の修正 → 関連する usecase への影響を確認
- core の修正 → 全レイヤーへの波及を確認（core/README.md の更新プロトコルに従う）

**2. 変更の粒度**

- 一度に変更するのは1つのファイル、1つの観点に限定する
- 複数の気づきがあっても、個別の更新として扱う
- 大きな変更は仮説として小さく試し、検証してから確定する

**3. 更新の記録**

- 何を変えたか（差分）だけでなく、なぜ変えたか（動機）を残す
- どの振り返りから生まれた更新かを追跡可能にする

### 更新の優先順位

複数の更新候補がある場合の優先順位。

1. 誤った判断を誘導するリスクがある定義の修正
2. 頻繁に遭遇するパターンの usecase 追加
3. 既存定義の精度向上（表現の改善、観点の追加）
4. 新規レイヤーの追加

### アンチパターン

- 気づきを溜め込んで一括更新する（鮮度が落ち、文脈を忘れる）
- 1回の失敗で原則を変える（パターンが確認されるまで待つ）
- 更新のための更新（システムをいじること自体が目的化する）
- 振り返りなしに更新する（思いつきの改変はシステムを劣化させる）
