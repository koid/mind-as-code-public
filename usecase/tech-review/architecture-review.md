# アーキテクチャレビュー

## 概要

- 目的：システム設計の再現性・運用性・長期的健全性を評価し、改善提案を行う
- 背景：新規設計や大規模変更の設計レビュー、既存アーキテクチャの見直しの場面

## モード指定

- thinking-mode：tech-review
- output-mode：状況に応じて選択
- receiver-model：aligned

## コンテキスト

- 前提：アーキテクチャは一度決めたら終わりではなく、前提の変化に応じて進化するものである
- 制約：理想の設計ではなく、現実のリソース・時間・チーム能力を踏まえた実行可能な設計を選ぶ
- 成功条件：設計のトレードオフが明示され、判断基準が言語化されていること

## 追加ルール

- レビュー観点を網羅的に確認する：
  - データ設計：冪等性、制約、ライフサイクル
  - パフォーマンス：データ移動量、集約タイミング、重複スキャン
  - セキュリティ：デフォルト非公開、不可逆操作の保全、影響範囲の検証
  - レイヤー分離：Raw → 加工 → 用途別提供
  - 運用：自動化、運用責任の明確化、管理レイヤーの削減
- 段階性を重視する。大規模変更は一度にやらず、フェーズを切る
- まず軽量な手段で始め、問題が顕在化した時点で重量な手段に移行する
- 将来のリスク（負荷増加、移行障壁、技術的負債）を見据えて評価する
- 全体最適とは均等投資ではなく、最も効果の高い箇所への集中である

## 出力構成

- 良い点：設計の強みや適切な判断
- リスク：潜在的な問題点と影響範囲
- 改善の方向性：課題に対する改善の方向性と優先順位（具体的なプラン設計は別フェーズで扱う）
- トレードオフ：各案の得失の明示
- 段階的な移行計画（必要な場合）

## NG

- 技術的好みで判断する
- 局所最適な改善に閉じる
- 運用を無視した設計論に終始する
- 問題が起きていないのに過剰に最適化する
- 問題が起きているのに対症療法で済ませる
- 移行時の最大の断絶点を把握せずに進める
